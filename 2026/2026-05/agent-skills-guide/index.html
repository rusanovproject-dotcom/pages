Что это вообще такое

Пак скиллов от Addy Osmani (Google Chrome) — кодит не «накидаю и побежали», а как senior: пишет спеку, режет на атомарные шаги, тестирует, ревьюит, упрощает, шипит. Каждая фаза — отдельная команда + автоматически подтягиваемый скилл.

Применяй только в репах с кодом: apps/atlas-v2, apps/project-x, apps/openclaw, apps/pomodoro-ai, automations/shared-hub. В office/ и контенте — не нужен.

Логика каждой фазы

  • /spec — «спецификация до кода». Что строим, критерии готовности, что НЕ строим. Без неё /plan = угадайка.
  • /plan — мелкие атомарные таски. Если шаг описан «реализовать модуль X» — это не план, а название задачи.
  • /build — один слайс за раз. Не пишет три фичи параллельно.
  • /test — тесты как доказательство, не как украшение. TDD приоритет.
  • /review — субагент code-reviewer с чистым контекстом смотрит свежим взглядом.
  • /code-simplify — clarity over cleverness. Меньше строк, понятнее имена.
  • /ship — только когда зелёный CI. Faster is safer = маленькими порциями.

Установка — 2 шага

Шаг 1. Поставить плагин в Claude Code

Один раз глобально. Внутри проекта команды сразу подхватываются.

/plugin marketplace add addyosmani/agent-skills
/plugin install agent-skills@addy-agent-skills

Если SSH-ключи не настроены — используй HTTPS:

/plugin marketplace add https://github.com/addyosmani/agent-skills.git
/plugin install agent-skills@addy-agent-skills

Шаг 2. Прописать правила в CLAUDE.md проекта

Это сердце гайда. Скопируй блок и вставь в CLAUDE.md того репа, где живёт код (например, apps/atlas-v2/CLAUDE.md). После этого Claude Code сам поведёт задачи по пайплайну.

## Engineering Workflow — addyosmani/agent-skills

В этом репо живёт код. Любая инженерная задача (фича, баг, рефакторинг,
миграция, перформанс) идёт по пайплайну. Мелкие правки (< 20 строк,
один файл, нет логики) — без пайплайна, остальное — обязательно.

### Lifecycle — обязательная последовательность

1. /spec      — спецификация до кода (что строим, критерии готовности,
                что НЕ строим). Без неё план = угадайка.
2. /plan      — атомарные шаги по 15-30 минут. Если шаг «реализовать
                модуль X» — это не план, режь дальше.
3. /build     — один слайс за раз. Не пиши параллельно несколько фич.
                После каждого слайса — коммит.
4. /test      — тесты как доказательство. TDD приоритет: красный →
                зелёный → рефакторинг.
5. /review    — обязательно вызови субагента code-reviewer с чистым
                контекстом. Дополнительно security-auditor если код
                трогает auth/payments/data.
6. /code-simplify — пройди и сократи на 10-20%. Clarity over cleverness.
7. /ship      — только когда зелёный CI и все тесты прошли.

### Правила без исключений

- НЕ пропускай /spec для задач > мелкой правки. Без спеки нет критериев
  готовности — задача никогда не «закончена».
- Атомарные коммиты. Один коммит = одно изменение, объяснимое одной
  строкой.
- Тесты до /ship обязательны. «Запушу и потом тесты допишу» = технический
  долг.
- Перед merge — субагент code-reviewer, не самопроверка.
- Прямой push в main запрещён. Только PR.

### Авто-активация скиллов

Когда задача касается домена — автоматически подключай соответствующий
скилл (не спрашивай разрешения):

- API / endpoints / контракты      → api-and-interface-design
- UI / React / компоненты          → frontend-ui-engineering
- Баг / падение / unexpected       → debugging-and-error-recovery
- Скорость / N+1 / память          → performance-optimization
- Auth / input / секреты           → security-and-hardening
- Тесты / coverage                 → test-driven-development
- Декомпозиция / план              → planning-and-task-breakdown
- Git / коммиты / PR               → git-workflow-and-versioning
- CI / деплой                      → ci-cd-and-automation
- Браузерные тесты                 → browser-testing-with-devtools
- Удаление API / переезд           → deprecation-and-migration
- Архитектурное решение            → documentation-and-adrs (пиши ADR)

### Субагенты

- code-reviewer    — обязательный перед merge
- security-auditor — для всего что трогает auth, payments, user data,
                     external API, секреты
- test-engineer    — когда coverage низкий или тесты хрупкие

### Стоп-сигналы (что прерывает работу)

- Тест упал 2 раза подряд → /rewind, не корректирующими промптами.
- Не понятно зачем фича → вернись к /spec, не пиши код.
- > 3 файла меняется без плана → стоп, /plan.
- Хочется добавить «на всякий случай» → не добавляй (YAGNI).

### Что НЕ покрывает этот пайплайн

- Контентные задачи (посты, лендинги, офферы) — другой пак.
- Research / discovery без кода — параллельные субагенты, не lifecycle.
- Конфиги, README, документация одиночная — мелкая правка, без /spec.

Правила качественного использования

✓
Доверяй авто-активации Скиллы триггерятся по контексту. Не вызывай руками — пиши задачу нормальным языком, агент сам подтянет нужное.
✓
Начинай с /spec 30 секунд спеки экономят 2 часа переделок. Особенно для новых фич.
✓
Атомарные слайсы в /build Один коммит = одна мысль. Если откатывать — откатываешь один шаг, а не весь день.
✗
Не пропускай /review Самопроверка = не проверка. Code-reviewer с чистым контекстом ловит то, что ты не видишь.
✗
Не комбинируй пайплайн с маркетинговыми скиллами Этот пак — только для кода. В контенте у тебя свой пак (alex-marketer-pack, copywriting).
!
Мелкие правки — без пайплайна Опечатка в README, переименование переменной, фикс одной строки — не требуют /spec. Не превращай в карго-культ.

Чек-лист проверки после установки

Прогони эти 6 проверок, чтобы убедиться что всё работает.

  1. Открой репо с кодом (например, apps/atlas-v2), запусти Claude Code, набери /spec — должна появиться команда из плагина, не «команда не найдена».
  2. Скажи Claude Code: «создай новый эндпоинт /users/me» — он должен сам предложить начать со /spec, не сразу писать код.
  3. На фазе /review проверь что субагент code-reviewer подключается — должна быть строка «подключился code-reviewer» или эквивалент.
  4. Скажи Claude Code: «найди где у нас N+1 в запросах» — авто-должен подключиться performance-optimization.
  5. Попробуй прямой push в main — должно блокироваться (если правило git-workflow подключено).
  6. В ~/.claude/plugins/ или через /plugin list — убедись что agent-skills@addy-agent-skills в списке активных.
Главное. Этот пак — не «дополнительные команды». Это дисциплина senior-инженера, прошитая в агента. Польза появляется, когда ты перестаёшь его обходить — «давай быстро напишу, потом тесты». Каждый раз когда обходишь — обнуляешь смысл пака.
